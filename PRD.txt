Product Requirements Document (PRD): PropSight AI
Project Name: PropSight AI (SG Rental Copilot)
Version: 2.0
Status: Phase 1 Complete · Phase 2 In Progress
Author: Backend SDE (AI Transition)

1. Executive Summary

PropSight AI is a browser extension designed to empower renters in Singapore
(specifically targeting the PropertyGuru platform). It bridges the gap between
rigid platform filters and complex user needs by:
  • Extracting hidden deal-breaker constraints from listing descriptions as
    visual tags (Phase 1 — shipped).
  • Enabling client-side tag filtering so users can narrow results by
    PropSight-extracted attributes (Phase 2a — next).
  • Converting natural language queries into precise platform filters and
    AI-ranked results (Phase 2b/2c — planned).

Core Value Proposition:
  • Transparency: Reveal hidden constraints (e.g., "no cooking," "landlord
    staying," "address registration") instantly.
  • Efficiency: Filter and search by attributes PropertyGuru doesn't natively
    support.
  • Intelligence: Convert vague natural language queries into precise filters
    and AI-ranked results.

2. User Personas

  • The Expat Professional: Needs to know if "Address Registration" is
    possible (legal requirement). Values privacy ("No Live-in Landlord").
  • The Chinese Tenant: Specifically looks for "Cooking Allowed" (Heavy vs.
    Light) and "Visitor Policy".
  • The Remote Worker: Needs "Quiet Environment" and "High Floor/Windy"
    attributes often buried in text.

3. Functional Requirements

─────────────────────────────────────────────────────────────────────────────
Feature 1: The "X-Ray" Tagger  ✅ SHIPPED
─────────────────────────────────────────────────────────────────────────────

User Story: As a user browsing a listing list, I want to see critical
attributes (Cooking, Owner, Toilet) as visual tags on the card so I don't
have to click into every page.

Requirements (all delivered):
  • Auto-Extraction: Content script detects listing cards on search results,
    map view, and detail pages via MutationObserver.
  • Dual Extraction Pipeline:
    - Client-side: regex rule-extractor (rule-extractor.ts) — zero latency,
      no API key needed.
    - Server-side: OpenAI GPT-4o-mini via /v1/analyze — higher accuracy,
      optional.
  • Tag Categories:
    🔴 Red Flags: No Address Reg, Live-in Landlord, Partition Room,
       No Visitors, No Cooking.
    🟢 Green Flags: Heavy Cooking OK, Private Bath, No Agent Fee,
       Address Reg OK, No Landlord.
    🟡 Caution: Light Cooking Only, Visitor Restrictions, Agent Fee,
       Gender Preference.
    🔵 Info: Room Type, Bathroom Type, Lease Term, Additional Flags.
  • Visual Injection: Colored pill-shaped tags injected below listing
    price/title, with "🔍 PropSight" branding.
  • Local Caching: chrome.storage.local with 24h TTL. Revisiting a listing
    shows tags instantly.
  • Description Fetching: Server-side scraper extracts og:description /
    __NEXT_DATA__ from listing detail pages when the content script cannot
    access the full description.

─────────────────────────────────────────────────────────────────────────────
Feature 2a: Tag Filter Panel  🔧 NEXT UP
─────────────────────────────────────────────────────────────────────────────

User Story: As a user viewing search results with PropSight tags, I want to
filter listings by tag values (e.g., "show only Cooking OK + No Landlord")
so I can instantly narrow down results without clicking into each listing.

Requirements:
  • Floating Filter Panel: A collapsible panel injected on the search results
    page, positioned to not obstruct PropertyGuru's native UI.
  • Filter Dimensions (derived from existing ListingTags):
    - Cooking Policy: All / Heavy OK / Light Only / No Cooking
    - Owner Occupancy: All / No Landlord / Landlord Lives In
    - Bathroom: All / Private / Shared
    - Room Type: All / Master / Common / Studio / Whole Unit
    - Address Registration: All / Allowed / Not Allowed
    - Gender Preference: All / Any / Male Only / Female Only
    - Visitor Policy: All / Allowed / Restricted / Not Allowed
  • "Hide Unknown" Toggle: Option to hide listings where a tag value is
    unknown, so the user only sees listings with confirmed information.
  • Real-time Filtering: Show/hide listing cards instantly (CSS display
    toggle on [data-listing-id] elements). No page reload.
  • Match Counter: Display "Showing X / Y listings" in the panel.
  • Persistence: Save the last-used filter state in chrome.storage.local
    so it survives page navigation within the same session.
  • Pure Client-Side: No backend changes required. Operates entirely on
    tags already extracted by Feature 1.

─────────────────────────────────────────────────────────────────────────────
Feature 2b: Smart Search (NL → PropertyGuru Filters)  📋 PLANNED
─────────────────────────────────────────────────────────────────────────────

User Story: As a user, I want to type "Find me a master room near Clementi
with no landlord and cooking allowed, under 4k" and get relevant results
without manually adjusting 10 filters.

Requirements:
  • Search Bar: An injected input box at the top of the search results page.
  • Intent-to-Filter: Backend converts user query into PropertyGuru URL
    parameters (district_code, maxprice, property_type, freetext, etc.)
    via LLM structured output.
  • Auto-Redirect: Extension redirects to the filtered PropertyGuru URL.
  • Tag Criteria Pass-through: The query also generates PropSight tag
    criteria (e.g., cooking=heavy_cooking_ok, owner=owner_not_staying)
    that auto-activate the Feature 2a Filter Panel on the results page.

─────────────────────────────────────────────────────────────────────────────
Feature 2c: AI Reranking + Top Picks  📋 PLANNED
─────────────────────────────────────────────────────────────────────────────

User Story: As a user who has run a smart search, I want to see the top
listings re-ordered by how well they match my specific needs, with an
explanation of why each listing was recommended.

Requirements:
  • Automated Fetching: Extension fetches and analyzes the top ~50 listings
    (2-3 pages) from the filtered results.
  • Semantic Reranking: Backend re-orders listings by semantic similarity
    to the user's original query using embeddings.
  • Top Picks Carousel: Inject a "PropSight Top Recommendations" section
    at the top of the results page, showing the top 5-10 matches.
  • Match Reason: Each recommended listing includes a short AI-generated
    explanation of why it matches.

4. Non-Functional Requirements

  • Latency:
    - Tags: <2s if cached, <5s if live extraction.
    - Filters: Instant (<100ms) — pure client-side DOM manipulation.
    - Smart Search: <3s for NL → URL conversion (Phase 2b).
  • Stateless Backend: The backend is a pure LLM proxy — no database,
    no Redis. All persistent caching is client-side.
  • Compliance: Must not trigger PropertyGuru's WAF/Anti-Bot protections.
  • Cost Efficiency: Minimize LLM token usage via local caching, batch
    processing, and client-side regex extraction as first pass.
  • Privacy: No user data stored on the server. Tags and filter preferences
    are only stored in the user's own browser.

5. UI/UX Guidelines

  • Minimalist: Do not clutter the existing interface. Filter panel is
    collapsible and positioned unobtrusively.
  • Native Feel: Use UI components (fonts, colors) that blend with
    PropertyGuru's design system.
  • Transparency: Clearly indicate which tags are AI-generated via
    "🔍 PropSight" branding on tag rows and filter panel.
  • Progressive Disclosure: Show essential filters first; advanced options
    (gender, visitor policy) can be collapsed.
